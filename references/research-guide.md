# 调研方法与搜索策略

本文档指导如何在产品设计各阶段进行有效的网络调研。调研不是独立阶段，而是贯穿全程按需触发的能力。

## 核心原则

1. **目的先行**：每次调研前明确"我需要什么信息来支撑当前步骤的决策"，避免漫无目的的搜索。
2. **压缩呈现**：调研结果应精炼为对产品设计有直接价值的摘要，而非大段转述原文。
3. **标注来源**：关键数据（市场规模、用户数等）应注明来源，方便用户判断可信度。
4. **承认局限**：网络搜索不能替代真实用户研究。调研发现的是公开信息和二手数据，应在PRD中标注哪些假设需要后续验证。

---

## 各步骤调研指南

### 阶段一 1.1：市场与用户调研

这是调研最集中的环节，目标是帮用户建立对市场和用户的基本认知。

**搜索策略：**

**第一轮：市场概况**
- 搜索词模板：`[行业/领域] 市场规模 趋势 [当前年份]`
- 搜索词模板：`[行业/领域] industry report [当前年份]`
- 关注点：市场规模（TAM/SAM/SOM）、增长趋势、主要驱动力

**第二轮：竞品分析**
- 搜索词模板：`[产品类型] 竞品 对比`、`[产品类型] alternatives`
- 搜索词模板：`best [产品类型] apps/tools [当前年份]`
- 对每个主要竞品进一步搜索：
  - 其官网了解定位和核心功能
  - 应用商店评价了解用户反馈
  - 搜索 `[竞品名] review` 了解深度评测
- 关注点：竞品定位差异、核心功能列表、定价模式、用户评价中的高频抱怨

**第三轮：用户痛点**
- 搜索词模板：`[场景描述] 痛点`、`[用户类型] 使用 [产品类型] 问题`
- 搜索社区内容：`site:zhihu.com [相关话题]`、`site:v2ex.com [相关话题]`
- 搜索词模板：`[产品类型] user complaints`、`[产品类型] pain points`
- 关注点：用户最常抱怨什么、现有方案的核心缺陷、未被满足的需求

**输出格式：**
```
## 调研摘要

### 市场概况
- 市场规模：[数据 + 来源]
- 增长趋势：[描述]
- 关键驱动力：[列表]

### 竞品格局
| 竞品 | 定位 | 核心功能 | 优势 | 不足 |
|------|------|---------|------|------|
| [名称] | [一句话] | [关键功能] | [强项] | [弱项] |

### 用户痛点
1. [痛点1]：[描述 + 证据来源]
2. [痛点2]：[描述 + 证据来源]

### 机会点
基于以上调研，以下领域存在产品机会：
1. [机会点1]
2. [机会点2]
```

### 阶段二 2.2：竞品功能对标

在功能收敛阶段，需要知道竞品已经普遍具备哪些功能（用于判定基本型需求）。

**搜索策略：**
- 搜索词模板：`[竞品名] features list`、`[竞品名] 功能介绍`
- 搜索词模板：`[产品类型] must have features`、`[产品类型] 必备功能`
- 搜索词模板：`[竞品A] vs [竞品B] comparison`

**输出格式：**
```
## 竞品功能对标

| 功能 | 竞品A | 竞品B | 竞品C | 判断 |
|------|------|------|------|------|
| [功能名] | ✓ | ✓ | ✓ | 基本型（全有） |
| [功能名] | ✓ | ✗ | ✓ | 期望型（多数有） |
| [功能名] | ✗ | ✗ | ✗ | 潜在差异化点 |
```

---

### 阶段二 2.3：信息架构参考

设计信息架构时，可以参考同类产品的组织方式。

**搜索策略：**
- 搜索词模板：`[产品类型] information architecture`、`[产品类型] navigation structure`
- 搜索词模板：`[产品类型] UX design case study`
- 如果有知名的同类产品，直接搜索其设计分析文章

**关注点：**
- 一级导航通常包含哪些入口
- 核心功能的层级深度（几步能到达）
- 有没有特别好的组织模式可以借鉴

---

### 阶段二 2.4：交互模式参考

设计用户流程时，遇到特定交互模式可以搜索最佳实践。

**搜索策略：**
- 搜索词模板：`[交互模式] best practice UX`
  - 例如：`checkout flow best practice UX`
  - 例如：`search filter UX pattern`
  - 例如：`onboarding flow design`
- 搜索词模板：`[交互模式] 设计规范`

**关注点：**
- 行业通行的步骤顺序
- 常见的反模式（应该避免的做法）
- 有数据支撑的设计决策（如"3步以内完成结账转化率最高"）

---

### 阶段三 3.1：边界场景参考

审查时发现不确定的边界场景，可以搜索行业做法。

**搜索策略：**
- 搜索词模板：`[场景描述] UX handling`
  - 例如：`empty state design patterns`
  - 例如：`error handling UX mobile app`
  - 例如：`concurrent editing conflict resolution`

---

## 调研质量自查

每次调研完成后快速自查：

- [ ] 调研结果是否直接回答了"我需要什么信息来支撑决策"这个问题？
- [ ] 是否压缩为对产品设计有价值的摘要，而非大段复述？
- [ ] 关键数据是否标注了来源？
- [ ] 是否标注了哪些是事实、哪些是推测？
- [ ] 调研结果是否自然融入了当前步骤的讨论中？
